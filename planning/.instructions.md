# planning/ — working plans and in-flight work

Tracked planning material: feature plans, refactor investigations, and their status. Distinct from `context/` (stable canon), `docs/` (the public site), and `plans/` (see below).

## `planning/` vs `plans/`

- **`planning/`** (this folder, committed) — the tracked planning surface. Start here.
- **`plans/`** (gitignored) — legacy/scratch working area: dated notes, licensed PHPP fixtures, and perf scratch referenced by `scripts/perf/`. Not committed, not the canonical planning home.

## Source-of-truth rules

1. **Plans are provisional; `context/` (and the authoritative `docs/` deep-dives) are canonical.** When a plan lands, fold the outcome into the right doc.
2. **`STATUS.md` is the index.** Every active feature/refactor gets a row.
3. **One folder per unit of work**, lowercase kebab-case.

## Layout

```
planning/
  README.md          # read order
  STATUS.md          # master index of active work
  features/          # new-capability plans   (features/<slug>/)
  refactor/          # cross-cutting refactors (refactor/<slug>.md or <slug>/)
  archive/           # completed/superseded    (archive/<slug>/ + README index)
```

## Feature/refactor folder contract

```
<slug>/
  README.md   # router: scope, read order
  PRD.md      # what/why
  STATUS.md   # current state, next step, blockers
  PLAN.md     # optional implementation sequence
```

For small items, a single `<slug>.md` is enough.

## Archiving completed work

When an item is `Complete`/`Superseded`: fold the outcome into `context/`/`docs/`, then **move** its folder to `planning/archive/<slug>/` unchanged and add a row to `planning/archive/README.md`. Flat by slug, never nested by date.

## Status vocabulary

`Requested` · `Scoped` · `In progress` · `Blocked` · `Implemented` · `Complete` · `Deferred` · `Superseded`.
